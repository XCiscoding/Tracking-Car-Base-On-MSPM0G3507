# 小车 Debug 报错日志

> 用于记录每一次调试中的报错、异常现象、判断、修复和验证结果。和 `experiment_log.md` 分工不同：实验日志记录测试数据；本文件记录 Debug 过程和排错决策。

## 记录字段说明

- 时间：发生或记录日期。
- 现象 / 报错：用户看到的错误、OLED 异常、Build Output 或车辆异常。
- 判断：当时认为最可能的原因。
- 处理：实际改了什么。
- 验证：改完后如何验证，是否通过。
- 后续：仍未解决时，下一步查什么。

## 2026-05-23｜Keil 编译错误：`DL_SYSTICK_getTickCounter` 未声明

### 现象 / 报错

Build Output：

```text
user_imu/imu.c(239): error: call to undeclared function 'DL_SYSTICK_getTickCounter'; ISO C99 and later do not support implicit function declarations
```

后续 Flash Download 也失败：

```text
*** error 56: cannot open file
Flash Download failed - Could not load file ... .axf
```

### 判断

`DL_SYSTICK_getTickCounter()` 不是当前工程可用的 DriverLib API；`.axf` 打不开只是编译失败后的连带结果，不是烧录链路问题。

### 处理

- 在 `keil/user_imu/imu.c` 中包含 `motor.h`。
- 把 `DL_SYSTICK_getTickCounter()` 改成项目已有接口 `Motor_GetTickMs()`。

### 验证

Keil 命令行构建通过，无输出错误。

## 2026-05-23｜IMU RAW 增长但 `C/OK=0`

### 现象 / 报错

OLED 显示 `RAW` 持续增长，但 `C` 和 `OK` 均为 0。

### 判断

UART1/PB7 接收链路是通的，问题不是接线完全不通，而是当前协议解析没有命中真实帧。

### 处理

- 把 OLED 改为显示最近 16 个原始字节。
- 通过用户回传原始字节确认真实帧头是 `55 55`。
- 将 IMU 解析器从 `AA FF CMD LEN DATA SUM` 改为 `55 55 CMD LEN DATA SUM`。

### 验证

`OK` 持续增长，`C` 在 `1/2/3/6` 之间切换。

## 2026-05-23｜传统 R/P/Y 不随水平转动变化

### 现象 / 报错

`OK` 已持续增长，但静止和手动水平转动车身时，传统 R/P/Y 基本只在 0~2 之间跳动。

### 判断

协议解析成功不等于姿态角可用。`CMD=0x06` 的传统 Roll/Pitch/Yaw 当前不能代表车体水平航向。

### 处理

- 暂停把传统 Yaw 接入 PID。
- 改 OLED 显示原始候选通道：`CMD=0x02` 的 G0/G1/G2 和 `CMD=0x03` 的 C0/C1。
- 手动测试正中、右转、回正、左转。

### 验证

确认 `D0 = CMD=0x02[0] - bias` 是当前唯一明显有水平转向响应且回正接近 0 的候选通道。

## 2026-05-23｜D0 积分导致 H 不回零、持续增大

### 现象 / 报错

用户反馈：

```text
D0 回到接近 0，但 H 不回零，而且还持续增大
```

### 判断

当前 G0/D0 不适合按纯角速度积分；如果 D0 停止后能保持偏移，更像某种融合/绝对通道或可直接用的误差量。继续积分会制造漂移。

### 处理

- `IMU_GetYaw()` 不再使用积分值。
- 改为直接返回 `IMU_GetHeadingGyroRaw() * -0.1f` 作为诊断显示。
- 后续直线控制直接使用 `IMU_GetHeadingGyroRaw()`，不走积分。

### 验证

手动测试显示 D0 能在回正后接近 0，并且左右转方向可区分。

## 2026-05-23｜D0 初次接入直线控制后偶发大偏

### 现象 / 报错

D0 直线测试初版数据：

```text
往左偏30°，531，3，6
往右偏5°，11，1，0
往右偏5°，16，3，0
```

### 判断

表现不稳定，用户怀疑“刚启动的时候两个轮子的初始脉冲不一样”。这个判断合理：起步瞬间编码器脉冲少、轮胎抓地未稳定，立即启用 EC/HC 容易放大扰动。

### 处理

加入启动保护段：

```text
前 250ms 或左右轮任一边脉冲 < 12：
  EC = 0
  HC = 0
  固定 PWM：左 170，右 220
过启动段后：
  EC + HC 都启用
```

### 验证

构建通过，但后续测试仍出现右偏 60°/15°，说明单纯关闭启动修正不够。

## 2026-05-23｜加入启动保护后仍主要右偏

### 现象 / 报错

测试数据：

```text
往右偏60°，603，2，6，7
往右偏15°，433，3，5
往右偏15°，455，2，5
几乎没有偏差，208，1，2
```

### 判断

右偏时 D0 为正几百，而旧控制方向会让左轮更快、右轮更慢，可能继续加重右偏。D0 修正方向大概率反了。

### 处理

- 反转 D0 修正方向：
  ```text
  left_pwm  = base + trim - EC - HC
  right_pwm = base + trim + EC + HC
  ```
- 加大 D0 修正力度：
  ```text
  HC = D0 / 80 → D0 / 40
  ```
- `HC` 最大仍限制为 ±50。

### 验证

构建通过；后续 4 次中 3 次基本可用，说明方向明显更对。

## 2026-05-23｜反转 D0 方向后仍偶发右偏 45°

### 现象 / 报错

测试数据：

```text
几乎没有偏移，9，6，0
往右偏45°，119，4，2
几乎没有偏移，44，4，1
往右偏10°，14，4，0
```

### 判断

D0 修正方向已经基本正确，但大偏那次停车时 `D0/E/HC` 都不大。说明问题可能发生在起步瞬间，而不是后半段 PID 修正太弱。

### 处理建议

下一版不要继续加 D0 增益，改启动段策略：

```text
0~250ms：启用编码器 EC，禁用 D0 的 HC
250ms 后：EC + HC 都启用
```

同时把 OLED 刷新从 500ms 改为 200ms，用来观察起步瞬间 `D0/E/EC/HC` 是否跳变。

### 验证

待下一轮修改和实测。
